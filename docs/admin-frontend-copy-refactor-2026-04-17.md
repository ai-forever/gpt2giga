# План рефакторинга: уменьшение текстовой перегрузки в admin frontend

Дата: 2026-04-17
Контекст: review текущего `/admin` UI по коду и по фактическим desktop-экранам
Область: `gpt2giga/frontend/admin/` + `packages/gpt2giga-ui/src/gpt2giga_ui/`

## Зачем нужен этот документ

Текущий admin frontend уже стал заметно лучше по структуре, но в интерфейсе всё ещё есть системная проблема:

- UI слишком часто объясняет сам себя;
- один и тот же смысл повторяется в нескольких слоях подряд;
- страницы выглядят длиннее и тяжелее, чем требуют реальные operator-задачи;
- справа и внизу часто появляется support-copy, который конкурирует с основным действием.

Это не проблема цвета или базового layout.
Основная проблема сейчас в copy-density и в повторяющихся шаблонах, которые автоматически делают каждую страницу многословной.

## Важная оговорка

Мобильные устройства не являются целевым сценарием для этого рефакторинга.

Значит:

- не нужно оптимизировать UX под phone-first flow;
- не нужно сохранять длинный mobile stack как обязательный сценарий;
- при принятии решений приоритет у desktop и laptop экранов;
- responsive можно держать только на уровне "не ломается", а не "полноценный mobile UX".

## Правило по коммитам

После каждой выполненной задачи и после каждого завершённого slice нужен отдельный commit.

Это обязательная часть процесса для этого refactor-направления:

- завершённая задача не должна оставаться только в working tree;
- progress-файл нужно обновлять после завершённого slice;
- после этого изменения нужно фиксировать отдельным commit перед переходом к следующей задаче.

## Главный диагноз

Сейчас перегруз появляется не из-за одной неудачной страницы, а из-за сочетания четырёх паттернов:

1. Shell уже содержит объясняющий контекст:
   - brand copy;
   - workflow-group copy в rail;
   - link meta в rail;
   - hero subtitle;
   - hero context copy.

2. Каждая страница добавляет ещё один уровень пояснений:
   - intro paragraph;
   - callout paragraph;
   - card intro;
   - table intro;
   - handoff intro.

3. Shared templates по умолчанию многословны:
   - `renderWorkflowCard(...)` всегда выводит `title + note + pills + actions`;
   - `renderGuideLinks(...)` почти всегда добавляет ещё один explanatory block;
   - `renderSubpageNav(...)` тоже содержит intro-текст.

4. Почти каждый информационный блок содержит не только значение, но и интерпретацию:
   - metric value;
   - note под metric;
   - definition note;
   - extra handoff sentence.

В сумме страница начинает не вести пользователя, а разговаривать с ним без пауз.

## Что в текущем UI уже хорошо

- Есть цельная визуальная система, а не случайный набор admin-виджетов.
- Цвет, глубина и панельная композиция работают хорошо для operator console.
- Основная информационная архитектура понятна: `Start`, `Configure`, `Observe`, `Diagnose`.
- Есть хороший page-driven split вместо одной сверхдлинной страницы.
- Raw diagnostics уже уводятся во вторичный слой через `details`, `code-block`, side panels.

То есть рефакторинг должен быть не про redesign, а про сокращение лишнего текста и выравнивание приоритетов.

## Цель refactor slice

Нужно сделать UI тише и короче без потери операторского контекста.

Ощущение после рефакторинга должно быть таким:

- страница быстрее читается сверху вниз;
- главное действие видно раньше;
- боковые блоки меньше спорят с main column;
- helper-text остаётся только там, где он реально помогает;
- guide/handoff логика остаётся, но перестаёт доминировать визуально.

## Основные принципы

### 1. Один смысл объясняется один раз

Если мысль уже выражена в hero, не нужно повторять её в первой карточке почти теми же словами.

Если страница уже явно называется `Traffic`, не нужно в трёх соседних блоках повторять, что "эта страница summary-first".

### 2. Нельзя объяснять каждый блок

Не у каждой карточки должен быть свой intro paragraph.
Во многих местах достаточно:

- короткого заголовка;
- 1 строки статуса;
- действия.

### 3. Notes должны стать исключением, а не стандартом

Сейчас notes встроены почти везде.
После рефакторинга note должен оставаться только там, где без него пользователь реально ошибётся или потеряет контекст.

### 4. Handoff важен, но должен быть компактным

Handoff между страницами нужен.
Но он не должен занимать половину правой колонки.

### 5. Guide links должны жить глубже

Большинство guide-блоков уместнее:

- в `details`;
- в footer card;
- на hub pages;
- или только на genuinely complex pages.

Не нужно тащить длинный guide-tail на каждую focused surface.

## Что именно сокращать в первую очередь

### Shell

#### Left rail

Сократить объём текста в rail:

- оставить workflow-group label;
- оставить названия entrypoints;
- убрать или сильно сократить `nav-group__copy`;
- убрать постоянный `nav-link__meta` для большинства ссылок;
- показывать расширенные пояснения только для active entrypoint или по explicit disclosure.

Цель:

- rail должен быть навигацией;
- а не ещё одной explanatory surface.

#### Hero

Сделать hero короче:

- оставить `eyebrow`;
- оставить page title;
- оставить одну короткую subtitle;
- убрать или резко сократить `hero-context__copy`;
- chips оставить, если они реально помогают ориентации.

Сейчас hero часто уже даёт достаточно контекста, а следующий блок повторяет то же самое другими словами.

### Shared templates

#### `renderWorkflowCard(...)`

Нужен более компактный вариант.

Предлагаемое изменение:

- поддержать compact mode;
- в compact mode рендерить только:
  - `title`
  - `pills`
  - `actions`
- `note` делать optional и использовать только в 1-2 карточках на странице, а не во всех подряд.

#### `renderGuideLinks(...)`

Нужно перестать использовать это как обязательный page tail.

Предлагаемое изменение:

- перенести в disclosure или collapse;
- на focused pages показывать максимум 1 guide entry;
- полный набор guide links оставлять на hub pages;
- intro у guide-блока сократить до одной строки или убрать совсем.

#### `renderSubpageNav(...)`

Intro у subpage nav почти везде можно убрать или сократить до одной короткой строки.
Сама secondary nav уже достаточно объясняет, что это набор дочерних страниц.

## Приоритетные страницы

### 1. Overview

Текущая проблема:

- hero объясняет страницу;
- `Gateway status` снова объясняет страницу;
- `Workflow handoff` снова объясняет маршрут по продукту;
- `Guide and troubleshooting` добавляет ещё один explanatory tail.

Что делать:

- сократить `overview-callout` до 1 короткого headline и 1 строки;
- у KPI убрать лишние notes там, где число говорит само за себя;
- в `Workflow handoff` оставить 2 карточки вместо 4 либо перевести часть карточек в compact mode;
- guide block свернуть в `details` или убрать с overview совсем;
- `Recent error handoff` оставить более сухим, без лишнего intro paragraph.

Цель:

- overview должен быть fast-scan dashboard, а не текстовый стартовый guide.

### 2. Setup hub

Текущая проблема:

- hub уже сам по себе полезен, но рядом живут progress, warnings, next step и step cards, каждая со своим объяснением;
- часть смысла дублируется между readiness/warnings и step cards.

Что делать:

- сохранить progress block;
- warnings оставить только как реальные warnings;
- `Next recommended step` сделать компактным CTA-блоком без длинного описания;
- step cards сократить: 1 строка описания максимум;
- если статус можно выразить pill-ами, не дублировать его в prose.

### 3. Settings hub

Это сейчас один из самых удачных экранов после прошлого refactor, но он всё ещё выиграет от сокращения.

Что делать:

- у `Persistence posture` убрать лишний explanatory paragraph;
- в entry cards оставить только 1 строку описания;
- `History` можно сделать ещё суше, ближе к list surface;
- не объяснять повторно, что focused pages лучше, чем одна длинная settings page: это уже видно из структуры.

### 4. Playground

Текущая проблема:

- form intro;
- section intros;
- handoff cards;
- bootstrap summary;
- request preview intro;
- lifecycle intro;
- output intro;
- transport intro;
- guide block.

Это один из самых перегруженных по copy экранов.

Что делать:

- оставить короткий intro только у формы целиком;
- убрать большинство intros внутри form sections;
- правую колонку сократить до одного `Smoke workflow` блока;
- `Current bootstrap posture` свернуть или упростить;
- `Request preview` и `Transport transcript` должны быть self-explanatory по заголовкам;
- `Guide links` убрать из первого экрана focused page.

Цель:

- Playground должен ощущаться как рабочий инструмент, а не как интерактивная инструкция.

### 5. Traffic hub и child pages

Текущая проблема:

- после split структура стала лучше, но copy всё ещё длинный;
- каждая lane-card объясняет, когда её открывать;
- inspector и guide-block продолжают добавлять длинный хвост.

Что делать:

- на hub оставить только один короткий page-level guidance block;
- lane cards сделать ближе к dashboard cards:
  - title
  - small sample
  - stats
  - CTA
- guide block на child pages убрать в disclosure;
- inspector intro сократить вдвое;
- не повторять на каждой странице, что logs нужны только после narrowing: это уже можно выразить через CTA label и page structure.

### 6. Keys, Providers, System, Files, Batches

Эти страницы уже не выглядят катастрофически, но им тоже полезен общий принцип:

- меньше educational prose;
- меньше `muted` абзацев перед уже понятным блоком;
- меньше повторов между summary, pills и note;
- guide-block только там, где реально сложный handoff.

## Что можно упростить в CSS

Рефакторинг не только про copy, но и про то, как этот copy визуально шумит.

### Предлагаемые правки

- уменьшить число мест, где используется мелкий `12px/13px` explanatory text;
- увеличить контраст между primary content и secondary helper text;
- сделать side panels чуть тише визуально;
- сократить vertical padding в некоторых stacked helper blocks;
- уменьшить число случаев, где длинный secondary text появляется по hover/focus в rail.

Важно:

не нужно делать интерфейс пустым.
Нужно сделать так, чтобы вторичный текст выглядел действительно вторичным.

## Что можно смело не делать

С учётом того, что mobile не является целью, не нужно тратить отдельный slice на:

- полноценный mobile-first rail redesign;
- сложную перестройку phone navigation;
- глубокую оптимизацию под узкие экраны;
- попытку сохранить весь explanatory content на mobile в красивом виде.

Достаточно, чтобы:

- layout не ломался;
- страница оставалась технически читаемой;
- основной desktop UX был приоритетом.

## Предлагаемый план внедрения

### Slice 1. Shared copy-density cleanup

Изменения:

- сократить shell copy в rail и hero;
- добавить compact mode для `renderWorkflowCard(...)`;
- упростить `renderGuideLinks(...)`;
- сократить intro в `renderSubpageNav(...)`.

Ожидаемый эффект:

- весь UI сразу станет тише без page-by-page redesign.

### Slice 2. Overview + Setup + Settings

Изменения:

- почистить hub pages;
- убрать очевидные дубли;
- сократить explanatory prose на dashboard-like экранах.

Ожидаемый эффект:

- входные страницы станут гораздо легче для первого сканирования.

### Slice 3. Playground + Traffic

Изменения:

- убрать длинные support tails на working pages;
- сократить правые колонки;
- спрятать вторичные guide blocks глубже.

Ожидаемый эффект:

- focused pages станут ощущаться как инструменты, а не как документация внутри UI.

### Slice 4. Follow-up polish

Изменения:

- пройтись по `Keys`, `Providers`, `Files`, `Batches`, `System`;
- дочистить остаточные verbose sections;
- выровнять тональность copy по всему продукту.

## Критерии готовности

Refactor можно считать удачным, если выполняются условия:

- на первом экране каждой ключевой страницы стало меньше текста;
- правая колонка перестала спорить с основной задачей страницы;
- guide/handoff логика сохранилась, но ушла на второй план;
- по скриншоту страницы легче понять, где primary action;
- UI всё ещё выглядит как operator console, а не как пустая dashboard-оболочка.

## Короткий итог

Проблема текущего admin frontend не в архитектуре и не в визуальной теме.
Проблема в том, что интерфейс всё ещё слишком подробно комментирует сам себя.

Следующий полезный refactor должен быть не structural-first, а density-first:

- меньше повторов;
- меньше explanatory copy;
- компактнее workflow cards;
- глубже спрятанные guides;
- приоритет desktop/laptop UX;
- mobile без отдельной оптимизации.
